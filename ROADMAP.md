## 今夜の改善 (2026-07-08 夜間作業)

テーマ「無骨→フレンドリー」。PoC を「動かして観察できる」道具として磨く:
1. `vsm runs` 新設 — Run 一覧をフォルダ手掘りせず一覧表示
2. `vsm status` の人間可読化 — 生 UUID の羅列をやめ、System の役割名・タスク内容・実際の状態を表示(完了しても "submitted" のままになる表示問題の解消)
3. `vsm submit` の進捗表示 — 最大 30 分無言だった実行中に、どのシステムが動いているかを表示
4. テストの環境非依存化 — discord.py が無い環境でも README のコマンドがそのまま緑になるように
5. `--help` の平文化 — reST 記法や要件番号の生表示をやめ、人間向けの説明+使用例に
6. (第2弾)主要コマンドのエラーメッセージ友好化(正しい書式例と次の一手を提示)と `vsm replay` の要約表示(生 JSONL を開かずにあらすじが読める)。適用状況は `_overnight_20260708/codex/nanihold2.result.md` を参照

なお「完了しても submitted と表示され続ける」問題の根本修正(`task_state_changed` イベントの実発行)は、イベントスキーマと多数のプロパティテストに波及するため夜間自動実装には不適と判断し、下の中期項目に送った。

- `task_state_changed` イベントの実発行による状態表示の根本修正(表示側導出は今夜実装済みの暫定解)


Nanihold OS を単体で拡張するのではなく、**LETHE / Eos で構築した基盤に接続する**のが本筋:
- S4(環境スキャン)の情報源として LETHE(個人データレイク)を使う — 「会社」が持ち主の実データを観測できるようになる
- Run 間の長期記憶を独自実装せず、LETHE への observation 書き込みで代替する
- S1_Worker の実行能力(コード実行など)は、Eos 側で確立する実行・承認パターンを流用して導入する

## 長期の方向性

「AI 自動会社」— 自律的に環境を観測し(S4)、方針を保ち(S5)、資源を配分し(S3)、実務を行う(S1)組織体。MVP スコープ外として保留した項目(FSX 数値最適化、公共性測定、共有剰余配分、再帰的サブ VSM、Web ダッシュボード)は、PoC の観測性が十分になり、実需要が固まってから着手する。

## やらないこと(現時点)

- Web UI ダッシュボード(MVP スコープ外の方針を維持)
- S1 の無制限な実行能力(サンドボックスと承認の設計なしには載せない)
