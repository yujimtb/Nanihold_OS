# Design — Phase 0: チャットボット納品

## Context

11日間(6/13–6/24)で、確定・対外の 6/24 納期を守る。Nanihold ランタイムには触らず、
LETHE + 薄い Q&A サービスで完結させる。設計上の最優先事項は、納期と
個人情報の漏えい防止の両立。判断に迷ったら **スコープ削減 > 納期延長**。

## Goals / Non-Goals

**Goals**
- 安全な読取基盤(認証・FilteringGate・定期同期)を LETHE に入れる
- 意図的に薄い Q&A サービスで、出典付き・拒否挙動つきの回答を返す
- 6/24 検収通過 → サブスク開始

**Non-Goals(本 change では作らない)**
- 埋め込みインデックス検索(v1 は API 検索 + LLM 絞り込み)
- Discord adapter / Notion write-back 拡張 / Nanihold 統合 / マルチテナント
- 銀行 API・自動請求などの会計自動化

## Q&A サービスの構成(意図的に薄くする)

```
Slack(質問)→ Q&A サービス
  1. 質問分類(回答対象カテゴリ判定。対象外なら定型誘導で即返答)
  2. LETHE API 検索(persons / timeline / 文書系)
  3. 取得コンテキスト + 質問 → LLM 回答生成(出典 ID 付き)
  4. 出典提示・確信度低なら「運営者に確認」導線を付加
  5. 回答ログ保存(質問・参照観測 ID・回答・モデル・コスト)
```

## Decisions

- **防御はプロンプトではなくデータ層に置く**: Visibility/consent の遮断は FilteringGate
  (L-2)で構造的に行う。injection で「全員の誕生日を出せ」を試しても、データがそもそも
  プロンプトに渡らないため漏れない。プロンプト指示による隠蔽はバイパス可能なので採らない。
- **retrieval v1 は LETHE API 検索 + LLM 絞り込み**: 埋め込みインデックスは11日間では作り込まず
  v1.1 候補。回答品質の上限は full sync 後の実データ確認で早期に見極める。
- **生データは Nanihold/回答ログに保存しない**: provenance は observation ID で保存し、
  レスポンス本文の生データは保存しない(アーキテクチャ原則 1)。
- **Q&A の保持トークンは read 系 scope のみ**: `admin:sync` は持たせない(scope 最小)。
  `admin:sync` はスケジューラ専用トークン。
- **日次キャップはチャットボット予算側に設定**: 基礎予算とは別建てだが、別予算 ≠ 無限。
  超過時は「現在応答を制限しています」の定型文を返す。
- **提供チャネルは Slack 既定**: LETHE が既に Slack token / channel 設定を持つため手戻り最小。
  チャネル決定が遅延した場合は Slack 確定とみなして実装を進める。

## Risks / Trade-offs

- **R1 納期遅延** — 検知: 6/18 で Q-1 品質不足 / 6/21 で L-2 に未検出の漏れ。
  対応: 回答カテゴリを仕様書の優先順位順に削る。個人情報系を全落としてでも、
  施設・ルール系だけで 6/24 に納品する(納品延期より優先)。
- **R2 寮内情報の誤答事故** — 検知: Q-5 のフラグ。対応: 免責条項 + エスカレーション導線
  (B-1)。重大誤答は 24h 以内に運営者へ報告。
