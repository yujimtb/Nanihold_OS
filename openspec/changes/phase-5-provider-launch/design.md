# Design — Phase 5: プロバイダー製品化

## Context

確定・対外の 2027/3 提供開始に向け、βの成果を課金可能な製品に変える。9月以降の本人稼働低下と
友人主担当化を前提に、観測層は「調整作業」化済み、本フェーズは製品化(価格・UI・配布・契約)に集中する。

## Goals / Non-Goals

**Goals**
- 原価ベースの価格設計
- 非エンジニアが日常運用できる操作面(N-26)
- self-host デプロイパッケージとドキュメント
- 課金が発生する提供開始(2027/3)

**Non-Goals(post-launch に送る)**
- FSX 数値最適化(REQ 14.1)・公共性測定(REQ 14.2)・共有剰余配分(REQ 14.3)
- テンポラル・インターフェース完全版
- **Lv3 自動化の限定開放**(提供開始時点では開放しない)

## Decisions

- **原価の3倍を下回らない**: サポート工数が読めない初期の防衛線。
- **seat 課金を主軸にしない**: 「人を減らすほど安くなる」誘因が観測系プロダクトの思想と矛盾するため、
  組織月額(規模帯別)+ 従量のハイブリッドを第一候補にする。
- **CLI/events.jsonl を運用者に見せない**: 日常運用(タスク投入・承認・構造確認・レポート閲覧)は
  すべて Web UI で完結させ、βの運営者にユーザビリティを判定してもらう。
- **法人化を提供開始のブロッカーにしない**: 3月の提供開始は個人事業主のまま可能(β延長線の契約形態)。
  法人化は判断基準(年間売上見込み・契約相手の法人格要求・責任限定の必要性)で別途判断する。
- **SLA はベストエフォートを明文化**: 過剰な可用性は約束せず、「落ちたら気づいて復旧する」運用前提を文書化。

## 価格設計の入力(§9.1)

- N-25 の org 別原価実績(LLM・ストレージ・推論バッチ)
- β提供価値の段(月1 Lv0 可視化 / 月2 Lv1 気づき / 月3 Lv2 承認付き書き込み)

## 提供開始 DoD(§9.3)

- 課金が発生する契約が1件以上
- オンボーディングが手順書どおり1日以内に完了する
- サポート窓口と SLA(ベストエフォートの明文化)が定義済み

## Risks / Trade-offs

- **R4 本人稼働低下**(確定的)→ 友人主担当 + 観測層の自動運用で吸収。本フェーズは製品化作業に集中
- **依存脆弱性・テナント分離の穴**→ B-5 セキュリティレビュー第2回で総点検(認証・テナント分離・
  シークレット・依存脆弱性・LETHE public-release-audit 拡充)
