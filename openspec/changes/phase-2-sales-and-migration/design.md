# Design — Phase 2: 営業開始と移設

## Context

Phase 1 の生存基盤の上で、収益を定常化させる。チャットボット移設は納期都合の副産物ではなく、
プロダクトの中核命題「既存組織のファジィな吸収」の最初の実証として扱い、Runbook を営業資料へ転用する。

## Goals / Non-Goals

**Goals**
- サービス Node 型商品の雛形(N-15)と LETHE 読取契約(N-16)
- 無停止のチャットボット移設と標準 Runbook(B-3)
- 受託の入口(S5 商品定義・受注導線)と効率化(N-17 再利用、N-18 動的分化)
- 8/31 収益 > 燃焼

**Non-Goals**
- 観測層・構造推論(Phase 3)
- マルチテナント実運用・β(Phase 4)
- LETHE 書き戻し L-9(Phase 3)

## Decisions

- **サービス Node は「終了しないこと」が正常**: Run は引き続き観測・会計の単位とし、
  サービス Node には起動/停止イベント・ハートビート・再起動ポリシー・リクエスト単位会計を持たせる。
  README の「永続的責任は Node、Run は実行の観測単位」モデルの初の実運用例として ADR を残す。
- **lethe_query は読取のみ・scope 限定**: network_scope を LETHE URL のみに許可。レスポンス本文の
  生データは Event_Log に保存せず、ToolInvocation 記録と派生結論のみ残す(アーキテクチャ原則 1)。
- **移設は無停止・切り戻し可能**: 並行稼働でゴールデンセットを両系比較し、切替後1週間は v1 を温存。
  先方が切替に気づかないことを成功条件にする。
- **商品定義を S5 ポリシーとして実装**: 定義外の受注は human review 必須(N-10 の適用対象に追加)。
  最低受注額・概算レンジは内規(対外非公開)。
- **動的分化はフォールバック付きで導入**: 固定フローを回帰テストとして固定してから、フラグで
  動的モードへ切替。問題なければデフォルト化し、固定フローはフォールバックとして残す。

## 商品定義(S5 ポリシー)

- 受ける: コーディング受託(codex/claude code で完結: Web アプリ・スクリプト・データ処理・既存改修)、
  LETHE 型の情報基盤構築
- 受けない: 24/365 SLA、ネイティブアプリ、決済・医療・法務等の高リスク、納期1週間未満の新規
- 価格: 案件見積もり制。最低受注額と概算レンジを内規として持つ

## Risks / Trade-offs

- **R4 9月以降の本人稼働低下**(確定的)→ 8月中に自社タスクの Nanihold 化を完了させ、
  以後を「調整作業」化。友人合流で補完
- **R5 友人の合流遅延(院試)**→ L 系を後ろ倒し可(β開始11月まで許容)。N 系は本人単独で先行
- **R10 受託が取れない**(検知: 8月中旬パイプライン空)→ SHIMOKITA 深掘り(v1.1・追加開発)に切替。
  サブスク + 1顧客深掘りでも 8/31 黒字は可能

## 移行 Runbook(6手順 / B-3)

1. 現状凍結 → 2. 並行稼働(ゴールデンセット両系比較)→ 3. 会計接続(customer_id 付き)→
4. 切替(Event_Log 記録)→ 5. 監視期間1週間(切り戻し可能に温存)→ 6. 撤去・トークンローテーション
