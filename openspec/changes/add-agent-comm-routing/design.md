# Design: add-agent-comm-routing

## Context

着信は `lethe-channel-bridge` を経て全て LETHE に取り込まれるが「誰宛か」の概念がなく、エージェント(Node / Pilot 実行主体)を名指した連絡を当人へ届ける経路も、エージェント自身が返信を書いて投入する経路もない。返信は `reply-approval@1` → ブリッジ send という配信面はあるが、起草面が欠けている。

オーナー要望(2026-07-21):「返信は自動生成ではなく、各エージェントに対して疎通する通知システムと返信システムにする」。本設計は通知(inbound routing)と返信(outbound authoring)の二本柱を、既存資産の上に最小の新規機構で載せる。

## 既存資産の再利用

```
着信  ─▶ lethe-channel-bridge(observation 取込・登録済みチャネル)─▶ LETHE
                                                                     │
                                                    [新規] inbound routing:宛先=個名 を解決
                                                                     ▼
                                    エージェント(node_id / pilot_id)へ通知配送
                                                                     │
                                                    エージェントが返信文を起草(自動生成しない)
                                                                     ▼
                          [新規] 帰属付き reply-draft@1 を card-queue へ投入
                                                                     │
                                            reply-approval@1(オーナー承認・既存)
                                                                     ▼
                            lethe-channel-bridge の reply()/send()(既存経路)─▶ 相手
```

- 名前プール: `_cutover_20260720_fable_activation/asset/Agent_name.csv`(130 行、カテゴリ[居/糸/器/水/木/天候/地]・規模 1-3・意味座標・日本語ローマ字[Toki/Aki/Irie/Nagi/Mio/Kumo 等]・英名・ラテン名・いいねフラグ)。
- 既存機構: LETHE card-queue(`reply-draft@1` → `reply-approval@1` → `send-record@1`)、Intercom ブリッジ、Nanihold の WorkItem / Execution / Node、Interface node(`node:owner-interface`)、pilot(`codex-coding-s1` 等)。

## 論点 1: 宛先規約(オーナー承認事項)

エージェント宛と判定する着信表現の候補。

| 案 | 規約 | 長所 | 短所 |
| --- | --- | --- | --- |
| A. 先頭「名前:」 | 本文冒頭が `Toki:` / `Aki:` 等 | 明示的・実装単純・チャネル非依存 | 通常会話の冒頭コロンと衝突しうる |
| B. メンション | `@Toki` 等プラットフォームメンション | プラットフォーム自然・誤爆少 | Discord/Slack のメンション ID とローマ字個名の対応表が要る |
| C. エイリアス表記 | 個名の別名を許容 | 呼び方の揺れに強い | エイリアス台帳の維持コスト、曖昧一致で誤配送 |

**推奨: A(先頭「名前:」)を主、B(メンション)を補**。A を一次規約とし、プラットフォームメンションが取れる場合は B を優先的に採用する。いずれにも合致しない着信は ACR-02 に従い特定エージェントへ配送せず、既存の LETHE 観測取り込みに留める(取りこぼしは誤配送より安全)。エイリアス(C)は初版では持たない。**最終確定はオーナー承認事項。**

## 論点 2: 配送形態(オーナー承認事項)

通知をエージェントへ届ける先の形態。

| 案 | 配送先 | 長所 | 短所 |
| --- | --- | --- | --- |
| 1. Ledger イベント | Nanihold Ledger に通知イベントを追記 | 監査自然(ACR-04 と直結)・疎結合・エージェント未起動でも滞留可 | エージェントが Ledger を能動 poll する必要 |
| 2. 実行中 Execution への注入 | 走行中 Execution の入力へ差し込む | 即応・文脈内で処理 | 対象が実行中でないと届かない・実行系改変が大きい |
| 3. 新規 WorkItem 起票 | 通知 1 件 = WorkItem | 既存実行系にそのまま乗る・粒度管理と整合 | 起票粗粒度・返信までのリードタイム・大量着信で WorkItem 氾濫 |

**推奨: 1(Ledger イベント)を基盤に、必要時 3(WorkItem 起票)へ昇格**。既定は Ledger への通知イベント追記とし(疎結合・監査自然・エージェント未起動でも滞留)、返信起草という作業単位が要るものは WorkItem へ昇格する二段構えを推奨する。2(Execution 注入)は「対象が実行中のときだけ届く」制約が強く、初版では見送り。**最終確定はオーナー承認事項。**

## 論点 3: 命名割り当ての運用(オーナー承認事項)

- 割り当てはオーナーの明示決定(ACR-01)。システムは自動割り当てしない。
- `Agent_name.csv` の `いいね` フラグが立つ行を優先候補として提示する運用を推奨(オーナー選好の反映)。
- カテゴリ・規模・意味座標は、エージェントの役割・階層(例: 規模の大小を Node の粒度に対応)を表す補助メタとして提示のみ行い、機械的な自動対応づけはしない。
- レジストリは個名 ↔ `node_id` / `pilot_id` を 1 対 1 で保持し、同一個名の重複割り当てを禁ずる。

## 監査(ACR-04)

- 通知配送: 着信 id → 宛先個名 → 配送形態(Ledger イベント id / WorkItem id)を receipt で連結。
- 返信帰属: `reply-draft@1` に書き手個名を帰属付与 → `reply-approval@1` → `send-record@1` を、応答先着信 id で貫通トレース。
- これにより「誰が誰宛の連絡に、誰の承認で返したか」を後から復元できる。

## スコープ外(明示)

- 返信文の自動生成、承認レス自動送信、名前プールの自動割り当て。
- `lethe-channel-bridge` の import / card-queue / send 契約の変更。
- LETHE 側 projection・承認 UI の実装。

## リスクと対応

- **誤配送**: 宛先規約の曖昧一致。合致しない着信は特定エージェントへ配送せず観測に留める(fail-safe)。
- **通知の滞留**: エージェント未起動時。Ledger 基盤なら滞留可、返信が要るものは WorkItem 昇格で可視化。
- **個名の枯渇・衝突**: 1 対 1 制約と重複禁止で衝突を防ぎ、プール(130 行)の消費はオーナー割り当てで管理。
