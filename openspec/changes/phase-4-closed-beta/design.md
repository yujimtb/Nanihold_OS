# Design — Phase 4: クローズドβ

## Context

外部組織で製品仮説を検証し、価格設計に足る metering を得る。ベータ先は個人情報感度が高い
教育・財団系のため、同意・プライバシーが事業の前提条件。β開始は3系統の AND 条件
(同意パッケージ × 観測層 MVP × マルチテナント)であり、同時進行させる。

## Goals / Non-Goals

**Goals**
- 同意・プライバシーパッケージ(対外文書化)
- 外部2〜3組織での観測→構造表示→提案ループ
- マルチテナント実運用(N-24)と org 月次原価(N-25)
- 社会的受容性の早期検証

**Non-Goals**
- 価格設計の確定(Phase 5 / 2027年1月)
- Lv3 自動化の開放(β期間中は不可)
- 非エンジニア向け UI の完成(N-26 は Phase 5)

## Decisions

- **明示オプトイン**: 個人同意は「観測対象チャンネルへの参加 = 同意」ではなく明示オプトインとし、
  L-7 が機械的に執行する。
- **受け入れ上限3組織**: サポート能力の上限。多いほど良いではない。
- **提供価値を段階提供**: 月1 = 可視化(Lv0)→ 月2 = 気づきレポート(Lv1)→
  月3 = 承認付き書き込み(Lv2)。**Lv3 はβ期間中は開放しない**。
- **3問目を最重視**: フィードバック定型3問のうち「不快・不安はなかったか」を製品の社会的受容性の
  早期警報として最優先で扱う。申告は最優先対応。
- **metering は N-1 の転用**: 受託期のコスト会計を org 単位の原価集計に転用し、価格設計の入力にする。

## 同意パッケージ4点(B-6)

1. データ取扱説明書(何を収集/どこに保存/誰が見られる/いつ消える)
2. 同意フォーム(組織同意 + 個人同意の2層、明示オプトイン)
3. β協定書(無償・期待協力・当方義務・期間3ヶ月・相互即時解約)
4. インシデント対応方針(漏えい・誤表示時の通知と対応)

研究コミュニティ(HUMAI)向けには「AI による組織構造の自動推定の実証研究」という枠組みを併記。

## 提供形態(オンボーディング)

- A: 当方ホスティング(自宅サーバー、org 別 SQLite 分離)
- B: 先方 self-host(docker compose、友人が支援)

チェックリスト: 同意取得確認 → トークン発行 → チャンネル選定 → 初回 sync → スキーマ設定(L-5)
→ 初回推論 → 運営者への結果説明会(30分)

## Risks / Trade-offs

- **R8 β先での個人情報懸念・炎上**(致命・事業仮説)→ 同意パッケージ先行・Lv3 非開放・即時削除権。
  週次3問目の申告を最優先対応
- **R2 誤表示**→ インシデント対応方針に沿って通知・修正

## Exit(12/31 β成立判定)

§8.4 の成功指標表:稼働2組織以上(3ヶ月継続)/ O5 参加70%以上 / 「実態に近い」過半 /
不快・不安の重大事案ゼロ / org 月次原価把握済み / 受託・サブスク収入継続。
