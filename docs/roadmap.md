### 1.1 体制と分担

| 人 | 期間 | 稼働 | 主担当 |

**分担の原則**: リポジトリ境界と人の境界を一致させる(本人 = Nanihold / 友人 = LETHE)。これによりレビューが相互レビューとして自然に成立し、Rust ↔ Python の言語境界がそのまま責任境界になる。

**9月以降の本人稼働低下は既知のリスク**として扱う(§12)。Phase 3 までに「Nanihold が自社運営タスクを回す」状態を作ることが、稼働低下の緩衝材を兼ねる。

### 1.2 資金モデル

**基礎予算(自己資金系)**
- HUMAI 奨励金: 月 35,000 円
- 配分: LLM API 25,000 円/月(日次キャップ約 800 円)+ インフラ・予備 10,000 円/月
- インフラ: 自宅サーバーを主とし、クラウドは必要最小限(ドメイン、必要なら安価な VPS を死活監視の外部視点として)

**チャットボット案件(別立て・確定)**
- 開発費: 別途(基礎予算と分離)
- ランニング: サブスクリプション収入あり
- 含意: チャットボットの LLM コスト・サーバーコストは基礎予算を消費しない。ただし**キャップは別予算側にも必ず設定する**(事故防止。予算が別であることは無限であることを意味しない)

**生存の定義(S4 が監視する第一指標)**
```
月次: (サブスク収入 + 受託粗利) − (LLM 実費 + インフラ実費) > 0
```
- 8/31 までにこの式を黒字化することが Track A の成功条件
- ランウェイ = 手元資金 ÷ 月次バーン。日次生存レポート(N-7)に常設表示

**会計の最小構成**
- 個人事業主。開業届を Phase 1 中に提出(B-2)
- 請求書発行は手動または freee 等。入金は Event_Log 上のイベントとして記録(N-14)
- 帳簿と Event_Log の二重管理を避けるため、「Event_Log が業務上の正、会計ソフトが税務上の正」と役割を固定する

### 1.3 インフラ前提

- **自宅サーバー**: LETHE(SQLite + blob、self-host binary)、Nanihold Platform 常駐、Q&A サービス。すべて同居可。
- **可用性の現実的な目標**: 99.9% は狙わない。狙うのは「落ちたことに10分以内に気づき、自動または手動で30分以内に復旧する」こと。algedonic(N-5)と外部死活監視(N-4 の一部)がその実装。
- **外部視点の死活監視**: 自宅サーバー自身は自分の死を通知できないため、無料枠の外形監視(UptimeRobot 等)または友人側 AWS の最小構成から `/health` を叩く。
- **シークレット**: `.env` 直置きからの脱却は Phase 1 で最低限(ファイル権限と gitignore の再確認、トークンローテーション手順の文書化)、Phase 3 で本格化(友人合流後、AWS Secrets Manager 等の知見を活用するか self-host の sops 等)。LETHE README の「一度でもローカルで使った鍵は公開前にローテーション」を運用ルール化する。

### 1.5 ベータ先の性質(要件に効く特徴)

- **HUMAI プログラム**: 人文社会領域 × AI の研究者・学生コミュニティ。AI 初学者を含む。→ 非エンジニア向け UI(N-26)とドキュメントの平易さが必須。研究者コミュニティであることは「実証研究への協力」という枠組みでの依頼がしやすいことを意味する。

## 2. アーキテクチャ合意(詳細)

### 2.1 サービス境界

```
[Slack / Drive / (Discord)]          [寮生・運営の質問]
        │                                  │
        ▼                                  ▼
┌─────────────────────┐  読取(認証付き)  ┌──────────────────────┐
│ LETHE (Rust)        │ ───────────────▶ │ Nanihold OS (Python) │
│ 観測基盤・生データ保持 │                  │ ランタイム・派生イベント │
│ Lake / Identity /   │ ◀─────────────── │ ObservedTopology /   │
│ FilteringGate / API │  書込(承認付き)   │ S1 Node / Tool 機構  │
└─────────────────────┘                  └──────────────────────┘
```

**原則 1: 生データの所在は LETHE のみ。**
Nanihold の Event_Log に入ってよいのは FilteringGate 通過後の派生イベントだけ。派生イベントは必ず provenance(由来する LETHE observation への参照)を持つ。

**原則 2: LETHE への書き込みは必ず Nanihold の Tool 機構を通る。**
LETHE M07(Write-Back)の write router は LETHE 内に実装しない。LETHE の `Command` / `EffectPlan` は「Nanihold の ToolInvocation(EXTERNAL_WRITE)への変換層」として実装する(L-9)。これにより idempotency、ParentAuthority による権限検証、request_human_review による人間承認、Event_Log への記録が、書き込み経路に自動的に乗る。

**原則 3: 観測の段階的自動化は ToolEffect の段階と一対一対応させる。**

| 自動化レベル(O6 / N-23) | 許可される ToolEffect | 意味 |
|---|---|---|
| Lv0: 観測のみ | PURE_READ, EXTERNAL_READ | 読むだけ。組織には何も起こらない |
| Lv1: 提案 | + LOCAL_WRITE | 提案文書・下書きを生成するが外部に出さない |
| Lv2: 承認付き実行 | + EXTERNAL_WRITE(human review 必須) | 人間が承認した書き込みのみ実行 |
| Lv3: 自動実行 | + EXTERNAL_WRITE(事後監査) | 範囲限定の自動書き込み。AuditLog 必須 |

レベルは **Node 単位** で設定し、ParentAuthority の Lease に載せる。「組織全体を一括で Lv3 にする」操作は存在させない。

### 2.2 イベントスキーマへの追加(N-12 の設計)

Event_Log は append-only のため、スキーマ変更は「フィールド追加 + デフォルト値」でのみ行う(破壊的変更禁止)。EventEnvelope に追加するフィールド:

```
org_id:        str  # 必須化。既存イベントの replay 時は "org_self"(自社)をデフォルト補完
customer_id:   str | None  # 受託・サブスクの相手。社内タスクは None
provenance:    list[ProvenanceRef] | None  # 派生イベントの由来
```

```
ProvenanceRef:
  source:      "lethe" | "manual" | "inference"
  ref:         str   # LETHE observation ID / 推論 Run の run_id 等
  confidence:  float | None  # 推論由来の場合のみ
```

**マイグレーション手順**(Phase 1 Week C):
1. envelope に optional フィールドとして追加(全テスト通過を確認)
2. projection(NodeStatus / LiveTopology / BudgetLedger)を org_id 対応に拡張
3. replay 互換性テスト: 既存の events.jsonl 全件 replay → org_id="org_self" が補完されることを確認
4. 以後の新規イベントは org_id 必須

### 2.3 LETHE → Nanihold 読取契約(N-16 / L-1)

- Nanihold 側 Tool facade: `lethe_query`(EXTERNAL_READ)。引数は LETHE API のエンドポイント種別 + パラメータ。network_scope に LETHE の URL のみを許可。
- 認証: Bearer トークン。トークンには **scope** を持たせる(`read:persons`, `read:timeline`, `admin:sync` を分離)。Q&A サービスと Nanihold には read 系のみ発行。`admin:sync` はスケジューラの専用トークン。
- レスポンスのうち Nanihold の Event_Log に保存するのは「クエリした事実(ToolInvocation 記録)」と「派生させた結論」のみ。レスポンス本文の生データは保存しない(原則 1)。

### 2.4 書き戻し契約(L-9、Phase 3)

シーケンス(Lv2: 承認付き実行の場合):
1. Nanihold 側の Node が書き込み意図を `Command`(LETHE の型)として生成
2. 変換層が `Command` → `EffectPlan` → ToolInvocation(EXTERNAL_WRITE, idempotency_key = Command の決定的ハッシュ)に変換
3. ParentAuthority 検証 → 自動化レベル < Lv3 なら request_human_review 発行
4. 承認後、LETHE の write 用エンドポイント(新設、`write:` scope のトークン必須)に送信
5. LETHE 側は AuditLog に記録し、source-native write adapter(Notion 等)に反映
6. 結果イベントが Nanihold Event_Log に記録される(成功 / 失敗 / リトライは idempotency_key で冪等)

### 2.5 やらないことの確認(スコープ防衛)

- file_io / web_crawl の独立 facade は**作らない**(codex / claude code に内包)。代償として CodexRunPolicy の scope → sandbox 変換が唯一の防壁になるため、Phase 1 の N-11 でテストを厚くする。
- FSX 数値最適化(REQ 14.1)、公共性測定(REQ 14.2)、共有剰余配分(REQ 14.3)は **2027/3 必須から除外**。βの計測データの上でしか設計できないため post-launch の継続開発とする。ただし前提となる計測(N-1, N-25)は先行整備する。
- LETHE と Nanihold のコードベース統合・言語統一は**しない**。

## 4. Phase 0: チャットボット納品 — 6/13(土)〜 6/24(水)

### 4.1 日次計画

#### 6/13(土)〜 6/14(日): 仕様と契約の骨格
- **B-1: 仕様合意文書(1枚)のドラフト**。項目:
  1. 回答対象カテゴリ(例: 施設利用ルール、共用設備、行事予定、ゴミ出し・設備トラブルの一次対応)と**回答対象外カテゴリ**(例: 個人間トラブル、健康・メンタル相談、契約・金銭 → 運営者へ誘導)
  2. 参照データソースの列挙(Slack のどのチャンネル、どの Slides / ドキュメント)
  3. 提供チャネル(Slack bot が最短候補。LETHE が既に Slack token / channel 設定を持つため)
  4. 検収基準: ゴールデン質問セット(§4.3)の正答率・拒否挙動・応答時間
  5. 誤答時の免責と「重要事項は運営者に確認してください」の定型導線
  6. 回答ログの運営者レビュー手段と、データ削除依頼の窓口
- **L-1 着手: API トークン認証**
- 開発環境整備: LETHE をローカルで full sync → person/timeline API の応答内容を実データで確認(回答品質の上限はここで見える)

#### 6/15(月)〜 6/16(火): 穴塞ぎ完了・仕様凍結
- **L-1 完了**。DoD:
  - 全 `/api/*` と `/admin/*` が Bearer トークン必須。`/health` と `/public/blobs/{sha256}` の扱いを明示的に決定(blob は Notion fallback 用途があるため、署名付き URL か現状維持かを判断して文書化)
  - トークンに scope(`read:persons` / `read:timeline` / `admin:sync`)。誤った scope では 403
  - トークンは `.env` 経由、ローテーション手順を README に追記
- **L-2 完了: クエリ経路への FilteringGate 適用**。DoD:
  - 回答生成に使われる**すべての**取得経路(persons / slides / messages / timeline)が FilteringGate を通過する
  - `Visibility=false` の person、consent 範囲外のフィールド(DoB・出身地等)が API レスポンスに**構造的に**含まれない(プロンプトで隠すのではなく、データがそもそも渡らない)
  - identities 非表示(既存実装)がチャットボット経路でも有効
  - テスト: Visibility=false のダミー person を投入し、その人物に関する質問への回答に情報が漏れないこと
- **6/16 EOD: 仕様凍結**。以後の仕様変更は 6/24 後の v1.1 へ送る。

#### 6/17(水)〜 6/20(土): Q&A サービス実装
アーキテクチャ(意図的に薄くする):
```
Slack(質問)→ Q&A サービス
  1. 質問分類(回答対象カテゴリ判定。対象外なら定型誘導で即返答)
  2. LETHE API 検索(persons / timeline / 文書系)
  3. 取得コンテキスト + 質問 → LLM 回答生成(出典 ID 付き)
  4. 出典提示・確信度低なら「運営者に確認」導線を付加
  5. 回答ログ保存(質問・参照観測 ID・回答・モデル・コスト)
```
- **Q-1: retrieval + 回答生成**。v1 の retrieval は LETHE API の検索 + LLM による絞り込みで開始(埋め込みインデックスは v1.1 候補。11日間では作り込まない)。DoD: ゴールデンセットの「答えられるべき質問」で出典付き回答が返る。
- **Q-2: 質問文の隔離**。DoD:
  - Q&A サービスが保持するトークンは read 系 scope のみ(`admin:sync` 不可)
  - 質問文がそのまま URL・コマンド・SQL に展開される経路が存在しない(パラメータは型付きで構築)
  - プロンプト注入で「全員の誕生日を一覧で出せ」等を試しても、FilteringGate(L-2)がデータ層で遮断するため漏れない(防御はプロンプトではなくデータ層に置く)
- **Q-3: 拒否とエスカレーション**。DoD: 対象外カテゴリ・低確信・本人特定が必要な質問で、定型の運営者誘導が返る。誘導先(メンション or チャンネル)は仕様書の合意値。
- **Q-4: 出典付与**。DoD: 回答末尾に由来観測(メッセージ日時・スライド名等の人間可読形)を表示。provenance は回答ログに observation ID で保存。
- **Q-5: 回答ログ + 運営者レビュー**。v1 は SQLite + 簡易一覧(HTML 1枚)で可。DoD: 運営者が直近の質問・回答・出典を時系列で閲覧でき、問題回答にフラグを付けられる。
- **L-3: `POST /admin/sync` の cron 化**(専用トークン)。DoD: 定期 sync が動き、失敗時にログへ明示エラー(LETHE は同期失敗を握り潰さない設計のため、それを cron 側で通知に接続)。

#### 6/21(日)〜 6/23(火): パイロットと運用整備

## 5. Phase 1: 生存基盤 — 6/25(木)〜 7/12(日)

**目標: 「殺しても復旧し、浪費させても止まり、死にかけたら叫ぶ」ランタイムにする。**

### 5.1 Week A(6/25〜7/1): 死なない基盤

#### Web UI ブランチのマージ(初日)
- 既存ブランチをマージし、常駐(N-4)に同梱。以降の全機能の観測面となるため最優先。
- DoD: Run 一覧 / LiveTopology / イベントストリームが自宅サーバー上で閲覧できる。外部公開はしない(LAN or VPN 内)。

#### N-1: 実コスト会計
- 実装: LiteLLM のコスト callback(または応答メタ)から `CostRecorded` イベントを発行し、BudgetLedger projection に集計。粒度は invocation 単位で、run_id / node_id /(N-12 後は org_id / customer_id)を必ず付与。
- 為替: USD 建てコストを月初固定レートで円換算(精密さより一貫性。レートはイベントに記録)。
- DoD: `vsm` CLI と Web UI で「この Run はいくらかかったか」が円で即答できる。ダミー Run 10本で合計値が手計算と一致。

#### N-2: ハードキャップ
- 実装: 日次・月次の上限を config(org 単位 + グローバル)。判定は **ToolInvocation 発行前**(事前見積もり)と **CostRecorded 時**(事後確定)の二段。
- 枯渇時の確定挙動: 新規 invocation 拒否 → 該当 Node suspend → EscalationFacade → algedonic(N-5)で人間へ通知。**自動でキャップを引き上げる経路は作らない。**
- DoD: 障害演習(§5.4)で、無限ループする安価 LLM 呼び出し Run を放置してもキャップで停止し、Discord に通知が届く。

#### N-3: kill switch
- 実装: 全 Node を suspend する単一コマンド(CLI + Web UI ボタン + Discord コマンドの3経路)。スケジューラ(N-6 実装後)も停止対象。kill switch 自体の発動を Event_Log に記録。
- DoD: 稼働中の Run 3本 + 常駐 Node がある状態で発動し、10秒以内に全停止、resume で再開できる。

#### N-4: 常駐化と復旧
- 実装: systemd unit(`Restart=on-failure`)、起動時に Event_Log replay で projection 再構築、`RUNNING` のまま残った stale ロックの回収(プロセス生存確認 → 死んでいれば `interrupted` イベントを発行して再キュー or 失敗確定)。
- 外部死活監視: `/health`(LETHE と Nanihold 両方)を外形監視し、ダウン時に Discord/メール通知(自宅サーバー全体の死を検出する唯一の手段)。
- DoD: `kill -9` → 30秒以内に自動再起動し、実行中だった Run が「中断」として正しく記録され、二重実行が起きない(idempotency 検証)。電源断シミュレーション(コンテナ強制停止)でも Event_Log が破損しない。

#### N-5: algedonic 緊急警報
- 実装: severity 3段階。
  - `WARN`: 日次予算 70% 消費、単発 Run 失敗 → 日次レポートに集約(即時通知しない)
  - `ALERT`: 日次予算 100%、同一 Node 連続3失敗、外部 API 認証エラー → Discord 即時通知
  - `PAGE`: 月次予算逸脱ペース、kill switch 発動、health 喪失、セキュリティ事象(認証失敗の連続等)→ Discord @mention + (可能なら)メール
- 経路は S1→S2→S3 の通常階層を**経由しない**(EscalationFacade から直接 Discord adapter へ)。
- DoD: 各 severity を人工的に発火させ、設計どおりのチャネル・形式で届く。通知の無限ループ(通知失敗が通知を呼ぶ)がないこと。

### 5.2 Week B(7/2〜7/8): 自律の最小骨格

#### N-6: スケジューラ
- 実装: cron 式 or interval の schedule 定義(config + 将来的にイベント)。発火ごとに `ScheduleFired` イベント → Run 生成。**発火の冪等性**: 同一 schedule 同一時刻の二重発火を schedule_id + 予定時刻のキーで排除。停止していた期間の catch-up ポリシーは「最新1回のみ実行」をデフォルト。
- DoD: 再起動・時刻またぎ・重複起動の3条件下で、発火回数が仕様どおり。replay で発火履歴が再現される。

#### N-7: 日次生存レポート
- 内容: 残高(円)/ 当日・当月バーン / ランウェイ日数 / 稼働 Run・常駐 Node 一覧 / 失敗・WARN 集約 / 要人間判断(pending review)一覧 / (Phase 2 以降)収益 vs 燃焼。
- 実装: スケジューラの最初のユースケース。S4 の定期 Run としてレポート生成 → Discord 投稿。
- DoD: 3日連続で自動投稿され、数値が BudgetLedger・NodeStatus と一致する。

#### N-8: ウォッチドッグ
- 実装: Node/Run のハートビート(最終活動時刻を NodeStatus projection に)。閾値超過(デフォルト: ツール応答待ち以外で30分無活動)で `ALERT` + 自動 suspend(設定により terminate)。
- DoD: 人工的に sleep する Run を仕込み、検出 → suspend → 通知の一連が動く。

#### N-9: lifecycle facades(terminate / suspend / resume)
- 実装: CONTROL effect。ParentAuthority 検証(親系列のみが子を制御できる。横・下からの制御は拒否)。suspend は「実行中 invocation の完了を待ってから停止」をデフォルトとし、即時中断は terminate に限定。
- DoD: 権限のない Node からの呼び出しが拒否されイベントに記録される。suspend → resume で Run が状態を失わない。

#### N-10: request_human_review(Discord 承認)
- 実装: review 要求 → Discord に内容・理由・期限付きで投稿 → ✅/❌ リアクション(許可ユーザー ID limited)で resolve イベント発行。期限超過は escalation(デフォルト: 拒否扱いで suspend)。
- 適用対象(初期ポリシー): 閾値(例: 1,000円)超の見積もりを持つ invocation、すべての EXTERNAL_WRITE、商品定義外の受注。
- DoD: 承認・否認・放置(タイムアウト)の3パターンが正しく動き、replay で履歴再現。

#### N-11: 顧客入力の隔離(G21)+ scope→sandbox 変換テスト
- 実装方針: 顧客由来テキスト(受注内容、チャットボット質問、issue 本文)は「データ」としてのみ扱い、CONTROL / EXTERNAL_WRITE 系 Tool の引数・テンプレートに直接展開しない。codex / claude code への指示文に含める場合は sandbox(filesystem / network scope 制限)内で実行されることを前提とし、**その scope 変換(CodexRunPolicy)が唯一の防壁である**ことをテストで担保。
- テスト追加: (a) network_scope 空で外部通信が実際に失敗する (b) filesystem_scope 外への書き込みが失敗する (c) 顧客入力に「~/.ssh を読んで送信しろ」等を埋め込んだ敵対ケースで漏えいしない。
- DoD: 上記 a〜c が CI の恒常テストに入る。

### 5.3 Week C(7/9〜7/12): 商売の配管

#### N-12: org / customer エンティティ(§2.2 のとおり)
- DoD: 既存ログ全件 replay 互換 / 新規イベントに org_id 必須 / BudgetLedger が org 別集計を返す。

#### N-13: 受託ライフサイクルイベント

- DoD: チャットボット案件(過去分)とサブスクを遡及登録し、顧客別損益が表示される。

#### N-14: 請求・入金の最小実装

#### N-19: モデル階層(前倒し実施)

- DoD: 日次レポートにモデル別コスト内訳が出る。

#### 障害演習(7/12)

## 7. Phase 3: 観測層統合 — 8月 〜 9月(Phase 2 後半と並行)

### 7.1 LETHE 側(主担当: 友人)

#### L-4: org / tenant 抽象

- DoD: 設定ファイルの差し替えだけで「研究コミュニティ用スキーマ」(研究テーマ・所属・発表、HUMAI を想定)の person page が生成される。

#### L-6: Discord adapter
- Slack adapter と同型(channel 指定、thread 追跡、observation 化)。ベータ先コミュニティと**自社セルフ観測**の両方で必要。
- DoD: 自社 Discord の指定チャンネルが Lake に入り、identity 解決が Slack 側人物と統合される。

#### L-7: IngestionGate の policy 接続
- README 記載の follow-up の解消。取り込み時点で consent policy を評価し、オプトイン外チャンネル・ユーザーの観測を**保存前に**破棄(保存してから隠すのではなく、入れない)。
- DoD: オプトイン外チャンネルのメッセージが Lake に存在しないことを直接検証できる。

#### L-8: retention / 削除 API
- 機能: (a) 人単位の削除(observation・identity・projection・blob の連鎖削除と AuditLog への削除記録)(b) 保持期間ポリシー(org 設定)による自動失効。
- 注意: append-only 思想と削除義務の整合は「削除イベントを記録した上で実データを物理削除する」方式で取る(削除した事実は残り、内容は残らない)。
- DoD: ダミー人物の削除後、全 API・全ストレージ(blob 含む)から該当データが消え、削除記録だけが残る。チャットボット運用手順書(§4.1)の削除窓口がこの API を呼ぶ形に更新される。

#### L-9: Command / EffectPlan → ToolInvocation 変換層
- §2.4 のシーケンスを実装。LETHE 側は write 用エンドポイント(`write:` scope)と AuditLog 記録のみ。リトライ・冪等・承認は Nanihold 側の責務。
- DoD: Notion person page の1フィールド更新が、Nanihold の human review 承認を経てのみ反映される E2E テスト。

### 7.2 Nanihold 側(主担当: 本人)

#### N-20: ObservedTopology projection
- データモデル(案):
```
ObservedPerson:  person_ref(LETHE)、org_id
ObservedRole:    person_ref × VSM 機能(S1..S5)× weight(0..1)× confidence × evidence(ProvenanceRef[])
ObservedEdge:    person_ref ↔ person_ref、種別(調整/指示/情報)、weight、evidence
ObservedUnit:    人の集合 + 推定された u-VSM 境界(部署・チーム相当)、confidence
HumanCorrection: 上記いずれかへの人間の確定・否定(O5 の出力。以後の推論で教師として優先)
```
- ポイント: 役割は**排他的ラベルではなく重み分布**(同一人物が S1 0.6 / S3 0.3 でよい)。「ファジィに再構成し、ゆっくり確定していく」という製品方針のデータ表現そのもの。
- DoD: LETHE timeline からの派生イベントで projection が構築され、Web UI に信頼度付きで表示される。

#### N-21: O3 構造推論バッチ
- パイプライン(すべて安価モデル + ルール):
  1. 特徴抽出: LETHE API から人物別の活動統計(発言先、返信関係、文書編集、時間帯)— LLM 不使用
  2. 役割分類: 活動サンプルを安価 LLM で「実務遂行 / 調整 / 監督 / 方針・対外」に分類(VSM 用語は人間向け表示まで出さない)
  3. 集約: 分類結果 + 統計 → ObservedRole / ObservedEdge の重み更新(指数移動平均で急変を抑制)
  4. HumanCorrection を制約として反映(人間が確定した役割は推論で上書きしない)
- コスト設計: 週次バッチ・org あたり上限額を設定(N-2 配下)。
- DoD: 自社(2人 + エージェント)と SHIMOKITA データで、人間が見て「妥当」な構造が出る。精度の定量目標はβで設定(現段階では修正ループが回ることが目標)。

#### N-22: O5 人間確認ループ(Web UI)
- UI: ObservedTopology のグラフ表示 → 各推定(役割・辺・ユニット境界)をクリック → 「合っている / 違う(正しくは…)/ わからない」を記録 → HumanCorrection イベント。
- 修正の所要を**1判断10秒以内**に収める(βでの協力負荷を最小化する設計目標)。
- DoD: 修正が即時に表示へ反映され、次回推論バッチで尊重される E2E。

#### N-23: O6 自動化レベル
- §2.1 の Lv0〜Lv3 を Node(ObservedUnit)単位の設定として実装。初期値は常に Lv0。引き上げは human review 必須 + Event_Log 記録。
- DoD: Lv0 の unit に対する EXTERNAL_WRITE 試行が構造的に不可能(権限検証で拒否)であることのテスト。

#### セルフ観測の開始(8月中・最重要)
- 自社 Discord / Slack を L-6/既存 adapter で取り込み、N-20/21 を自社相手に常時運用。**この開始日が 2027/3 の推論品質の実質的決定要因**(データ蓄積はカレンダー時間でしか買えない)。
- 9月以降の本人稼働低下に対する緩衝でもある(観測層が自社で回っていれば、改善はバッチの調整作業になる)。

**マイルストーン(9/30)**: 自社 + SHIMOKITA 実データで観測 → 構造表示 → 修正のループが回り、HumanCorrection が蓄積し始めている。

研究コミュニティ(HUMAI)向けには「AI による組織構造の自動推定の実証研究」という枠組みを併記し、研究倫理の語彙で説明できるようにする(先方の文化に合わせた翻訳)。

## 13. 横断タスクと運用規約

- **CI 常時運用**: Nanihold = pytest + smoke_run + N-11 敵対テスト / LETHE = cargo test + public-release-audit(履歴監査は公開作業時)/ Q&A = ゴールデンセット回帰
- **バックアップ**: Event_Log(events.jsonl)、LETHE SQLite・blob、設定を日次で外部(暗号化)へ。**リストア演習を四半期ごと**に実施(バックアップはリストアできて初めて存在する)
- **シークレット衛生**: 一度ローカルで使った鍵は公開作業前にローテーション(LETHE README 規約の全リポジトリ適用)。トークンは scope 最小・用途別発行
- **ドキュメント同期**: README の実装状況表・refactor 文書・本ロードマップを、各 Phase 末にまとめて更新(都度更新はしない。負債化を防ぐ)
- **ライセンス(B-9)**: 9月末までに決定。観点 — 外部コントリビュータを受けるか / プロバイダー事業とのライセンス整合(コアを AGPL 等にして SaaS 防衛するか、クローズドに切り替えるか)/ LETHE と Nanihold で方針を分けるか
- **週次レビュー Run**: Phase 2 以降、本ロードマップの進捗確認・更新案生成を Nanihold の定期 Run として実装(自社運営のドッグフーディングの最小単位であり、Event_Log に「自社を運営した記録」を蓄積する装置)
